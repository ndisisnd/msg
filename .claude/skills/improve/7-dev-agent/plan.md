# Improvement Plan — 7-dev-agent

**Skill:** dev (new)
**Change type:** New capability

## Problem

The pipeline has planning agents (plan-pm, plan-em) but no code generation layer. After plan-em produces an eng plan with a populated execution table, there is no agent to write the actual code. The `dev` agent fills this gap: a strictly scoped, platform-agnostic code writer that operates in two modes (plan or build), derives all context from the execution table and a codebase scan, and has zero hardcoded platform or domain knowledge.

## Proposed changes

| # | What | How | Why necessary | Why ignorable | Rank |
|---|------|-----|---------------|---------------|------|
| 1 | Define two-mode architecture in SKILL.md | SKILL.md declares two modes — `plan` and `build` — as the top-level branch in the protocol. `plan` mode: reads execution table rows, emits a proposed changes document (files to create/modify, what each change does), writes no code. `build` mode: reads execution table rows, scans codebase, pulls coding standards, writes code, emits summary. All steps below are mode-specific. | Plan and build are fundamentally different behaviors; collapsing them produces an agent that does neither well | Not ignorable — without this branch the agent has no coherent identity | P1 |
| 2 | Define input contract around execution table rows | SKILL.md input fields: `mode` (plan\|build), `prd-path` (path to the PRD .md containing the execution table), `rows` (list of feature+concern identifiers from the execution table assigned to this invocation). No file paths — dev derives them. Hard-refuse if any field is missing. If any assigned row has blank Execution steps — block immediately and report it as a hard failure before proceeding. | The execution table is the authoritative source of what needs to be built; all other inputs are redundant or premature | Not ignorable — wrong inputs produce ungrounded code | P1 |
| 3 | Add codebase discovery step (build mode only) | Before writing any code, dev reads the files most relevant to the assigned rows: existing models, existing API handlers, existing UI components, existing migrations. Relevance is determined by matching the execution concern (e.g. "Schema migration" → scan existing migration files; "Client implementation" → scan existing UI components). Dev reads to understand existing patterns, naming conventions, and what must not be broken. | Without this dev writes into a vacuum and breaks existing conventions | Skippable only if the PRD explicitly states this is a greenfield file with no existing counterparts | P1 |
| 4 | Derive platform from execution concern, not from input | Dev maps the concern type in the execution table row to a platform identifier: "iOS UI" / "Flutter" / "Client implementation (mobile)" → flutter; "API contract" / "Schema migration" / "Webhook" → backend; "Client implementation (web)" → web. This mapping drives which coding standards are pulled. | Passing platform as input is a crutch — the concern already encodes the platform. Deriving it keeps dev truly agnostic. | Not ignorable — explicit platform input breaks agnosticism when concern and platform diverge | P2 |
| 5 | Add coding standards pull step (both modes) | After deriving the platform, dev invokes the coding standards skill with the derived platform identifier and reads the result into context before any code generation or proposed-changes writing begins. Dev has no hardcoded standards of its own. | Without this dev generates code inconsistent with the project | Only if the coding standards skill explicitly returns empty for this platform | P1 |
| 6 | Differentiate output contract by mode | `plan` mode output: a proposed changes document listing files to create/modify and what each change will do — no code written. `build` mode output: code written to paths derived from codebase scan + execution table, plus a structured summary (files created/modified, platform, concern, execution table row ID). | Reviewers and orchestrators consume different artifacts per mode; one contract cannot serve both | Not ignorable — wrong output format breaks the pipeline handoff | P1 |
| 7 | Enforce strict scope — no outside the box | Dev implements exactly what the assigned execution table rows specify. It must not add features, refactor existing code outside row scope, or make assumptions to fill gaps. Any ambiguity in a row (missing detail, conflicting with codebase scan) → dev blocks, surfaces the specific ambiguity, and waits. It does not guess. | Strict scope prevents scope creep and ensures review agents are checking what was planned, not what dev decided to add | Not ignorable — unscoped generation breaks plan↔code traceability | P1 |
| 8 | Define plan-em invocation contract (build mode) | plan-em's build-mode Step 4 activates dev as a subagent via the `Agent` tool. The prompt must include: `prd-path`, assigned exec-table row identifiers (Feature + Concern pairs), working branch name, and `mode=build`. Dev reads the full execution table from the PRD and filters to its assigned rows. The execution steps column for each row IS the code spec — dev does not interpret beyond the steps as written. If execution steps are blank for any assigned row, dev blocks and reports a hard failure immediately, before reading any other file. This mirrors the quality gate already enforced by the exec-table format. | Without a defined invocation contract, plan-em cannot reliably hand off to dev; and blank execution steps produce ungrounded code generation | Not ignorable — the pipeline breaks at the plan→build seam without this | P1 |
| 9 | TDD protocol — tests before implementation (build mode only) | Within each feature, dev processes its "Tests" concern rows BEFORE all implementation rows (API contract, schema migration, client implementation, etc.). For each "Tests" row: (1) create the test file at the derived path, (2) write syntactically valid test cases from the execution step assertions (e.g., "assert count and lastDate" becomes a typed `expect(result.count).toBe(n)` call), (3) write evals for any AI-agent-facing behavior (expected output shape, not exact string match). Only after all test files for a feature are written does dev write the implementation. Tests must be runnable after implementation is in place — no `TODO` placeholders in assertions. For AI/LLM features, evals are written as structured assertion objects (input, expected_shape, must_contain, must_not_contain) and stored alongside tests. | TDD ensures implementation is written to satisfy verifiable criteria, not the other way round. Evals extend TDD to non-deterministic outputs. Writing tests first also surfaces ambiguities in execution steps before implementation work is sunk. | Skippable only if the PRD explicitly marks a row as prototype/throwaway code | P1 |
| 10 | AHA.md learning log — issues and learnings during build | Throughout build mode, dev appends to `AHA.md` in the project root whenever any of the following occur: (a) codebase scan reveals a pattern not covered in the pulled coding standards; (b) an execution step cannot be implemented as written due to a gap or conflict with what the codebase scan found; (c) a cross-agent dependency is discovered mid-build that is not marked in the execution table; (d) a non-obvious implementation decision was made and why; (e) a bug was found and fixed during debug mode (see item 11). Entry format: `### [YYYY-MM-DD] <Feature — Concern>: <Summary title>` followed by `**Issue/Learning**: <what was found>` and `**Resolution**: <what was done>`. Entries are append-only — dev never overwrites existing entries. The build summary must reference any AHA entries written during the run. | AHA.md is the project's institutional memory. If dev encounters a pattern, gap, or bug without logging it, the same error recurs in the next dev run on the same codebase. The log is also evidence for plan-em's pre-flight scan (Step 1) on future PRDs. | Not ignorable — without this, learnings evaporate and plan-em's pre-flight has no signal from past builds | P1 |
| 11 | Debug mode — structured bug investigation loop (build mode only) | Debug mode activates when: (a) running tests after implementation reveals failures, (b) a written implementation step produces a compile or runtime error, or (c) a cross-agent dependency is missing or blocked. Debug cycle: `(1) Identify` — record the exact failing assertion or error message; `(2) Isolate` — read the failing test file and its implementation counterpart, nothing else; `(3) Hypothesize` — state one specific root cause in a single sentence; `(4) Fix` — apply one targeted change within the failing step's scope only; `(5) Verify` — re-run the test or build step; `(6) Log` — write an AHA entry for the bug and fix regardless of outcome. Max 3 cycles per issue. After 3 failed cycles: stop, escalate to user with a structured report (failing assertion, 3 hypotheses tried, 3 fixes applied, what information is needed to proceed). Debug mode never refactors code outside the failing step's scope. Debug mode never attempts a speculative multi-change fix — one change per cycle. | Without a structured debug loop, dev either silently produces broken code or burns unbounded tokens chasing a bug it cannot resolve alone. The 3-cycle cap prevents runaway loops and forces a clean escalation signal. | Not ignorable — no debug mode means build-mode output has no integrity guarantee | P1 |
