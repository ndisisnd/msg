# Improvement Plan — 13-quality-mode-rigor

**Skill:** review (Quality mode)
**Change type:** Bug fix + new capability (three independent corrections to the Quality-mode contract)

## Problem

Quality mode is the first stage of the `/review` pipeline and is positioned as the gate for "structurally unsound" code. Three defects undermine that role:

1. **Stated checks have no backing flag.** [refs/modes/quality.md:5](../../review/refs/modes/quality.md) promises "code complexity, naming conventions, maintainability, dead code." None of those map to a `/cook` flag. The authoritative inventory at [FLAG-LIST.md](../../../../../cook/FLAG-LIST.md) confirms the eight global concerns are `--api-design`, `--architecture`, `--auth`, `--cicd`, `--debug`, `--error-handling`, `--performance`, `--security` — none cover dead code, duplication/DRY, readability, naming, or complexity, and no domain sub-ref does either. So the doc lies: Quality mode silently does not check what it advertises.

2. **Scope-creep signal is computed but discarded.** Step 4 of [SKILL.md:72](../../review/SKILL.md) emits `uncovered_changes[]` — files touched by the diff that don't map to a PRD row. Quality mode never receives this list, so additions outside the PRD's scope ("don't add features beyond what the task requires") are never flagged as findings. The signal dies in the surface object.

3. **Parallel agents produce duplicate findings.** Quality mode spawns 4 global + N domain `/cook` agents in parallel, all reading the same diff. Two agents will routinely flag the same `(file, line)` under different rule-ids (e.g. `--architecture` and `--api-design` both calling out a leaky boundary). The schema at [refs/schema.md:5](../../review/refs/schema.md) has no de-duplication contract, so the merged `findings[]` array inflates noise and biases the worst-verdict aggregation.

## Out of scope

- **Severity reweighting.** User is uncertain whether architectural findings should auto-promote to `block`. Defer until we have data on real Quality-mode runs.
- **Adding new `/cook` global concerns** (e.g. `--cleanup`, `--maintainability`). Cross-repo change to `vocab/tag-vocabulary.json` + `standards/global/refs/` + regenerating [FLAG-LIST.md](../../../../../cook/FLAG-LIST.md). Tracked separately if the inline rubric in change #1 proves insufficient.
- **Changing Coverage / Functional / Security / Performance modes.** This plan touches only Quality mode and the shared `findings[]` schema.
- **Changing `/cook`'s flag-resolution behavior.**
- **De-duplication across modes** (e.g. Quality and Security both flagging the same line). Only intra-mode dedup is in scope; inter-mode overlap is rare and informative.
