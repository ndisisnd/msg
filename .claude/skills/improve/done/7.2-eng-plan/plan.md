# Improvement Plan — 7.2-eng-plan

**Skill:** eng — plan mode (new)
**Change type:** New capability
**Part of:** `.claude/skills/eng/SKILL.md` (three-mode skill; see 7.1-eng-build and 7.3-eng-review for the other modes)

## Problem

After plan-em produces the execution table skeleton (Feature + Agent pre-populated, Execution steps blank), subagents fill in their execution steps — but there is no agent to translate those completed steps into a concrete, human-reviewable list of proposed file changes before any code is committed. Plan mode fills this gap: eng reads the assigned exec-table rows, scans the codebase, and emits a proposed changes document (which files to create or modify and what each change will do) for human approval before build begins.

**Artefact to create:** `.claude/skills/eng/SKILL.md` (plan mode section)

## Proposed changes

| # | What | How | Why necessary | Why ignorable | Rank |
|---|------|-----|---------------|---------------|------|
| 1 | Input contract for plan mode | Input fields: `--plan`, `prd-path` (path to the PRD .md containing the execution table), `rows` (list of Feature+Concern identifiers assigned to this invocation). Hard-refuse if any field is missing. Eng derives file paths; it does not accept them as input. Unlike build mode, blank Execution steps are not a hard failure in plan mode — they are surfaced as a gap in the proposed changes document. | The execution table is the authoritative planning input; missing fields leave eng without a scope boundary | Not ignorable — wrong inputs produce ungrounded proposed changes | P1 |
| 2 | PRD summary and user approval gate | After input validation, eng reads the assigned PRD section and exec-table rows, then emits a 3–4 line summary covering: (1) what the feature is — one sentence naming the feature, its user-facing purpose, and the scope of the assigned rows; (2) how to achieve this with code — 2–3 sentences on the layers to be touched, the main structural change, and key dependencies. Presents via `AskUserQuestion`: Approve and proceed / Needs correction / I have questions. If "Needs correction" — rewrites once and re-asks. If "I have questions" — enters user interview (item 7) then re-presents. Never proceeds without explicit approval. | The summary confirms eng has read the PRD correctly before it proposes file changes; a scope misread corrected here is cheaper than one corrected after a build | Not ignorable — wrong scope produces a useless proposed changes document | P1 |
| 3 | Codebase scan for proposed changes | After the approval gate, eng reads files relevant to the assigned rows to understand what exists: existing models, API handlers, UI components, migrations. Relevance matched by concern type (same logic as build mode's codebase discovery). The scan determines: which files already exist and would be modified, which do not exist and would be created, and what existing patterns constrain the implementation approach. | Without the scan, the proposed changes document is speculative and cannot name actual file paths | Skippable only if the PRD explicitly marks the row as greenfield with no existing counterparts | P1 |
| 4 | Derive platform from execution concern | Maps concern type to a platform identifier: "iOS UI" / "Flutter" / "Client implementation (mobile)" → flutter; "API contract" / "Schema migration" / "Webhook" → backend; "Client implementation (web)" → web. Drives which coding standards are pulled. Eng never accepts platform as input. | The concern already encodes the platform; accepting it as input breaks agnosticism when they diverge | Not ignorable | P2 |
| 5 | Coding standards pull | After deriving the platform, eng invokes the coding standards skill and reads the result before producing any output. No hardcoded standards. | Without standards, the proposed changes document may describe an approach incompatible with project conventions — wasted planning | Only if coding standards skill returns empty for this platform | P1 |
| 6 | Plan output contract — proposed changes document | Eng emits a structured proposed changes document. For each assigned exec-table row, the document lists: (a) files to create — path, purpose, what it will contain; (b) files to modify — path, what section is affected, what changes will be made; (c) any execution step that cannot be satisfied by a file change (e.g. a missing dependency) — flagged as a gap. No code is written. The document is the handoff artifact for human approval before build begins. | Reviewers and orchestrators need a concrete, file-by-file plan to approve before any code is committed; a vague description is not reviewable | Not ignorable — without a concrete output contract the plan mode produces no usable artifact | P1 |
| 7 | Strict scope enforcement | Eng proposes changes only for what the assigned exec-table rows specify. It must not propose additional refactors, unrelated file touches, or changes outside row scope. If a row is ambiguous, eng surfaces it as a gap in the proposed changes document rather than resolving it by assumption. | Out-of-scope proposals mislead reviewers and corrupt build mode's scope baseline | Not ignorable | P1 |
| 8 | User interview for unclear requirements | Eng may use `AskUserQuestion` proactively at two moments: (a) when the user selects "I have questions" at the approval gate — asks up to 3 focused questions, one at a time; (b) mid-protocol when a requirement is genuinely ambiguous and cannot be resolved from the PRD, exec-table, or codebase scan. Every question names the specific row or PRD section and offers 3–4 concrete options — no open-ended questions. Max 3 questions per invocation; if more than 3 ambiguities exist, surfaces them as a list and asks which to prioritise. After the interview, resumes protocol from where it paused. | Ambiguity resolved before the document is produced is cheap; ambiguity discovered after build starts is expensive | Ignorable only if PRD and exec-table are fully unambiguous | P1 |
