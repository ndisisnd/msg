# Improvement Plan — 7.1-eng-build

**Skill:** eng — build mode (new)
**Change type:** New capability
**Part of:** `.claude/skills/eng/SKILL.md` (three-mode skill; see 7.2-eng-plan and 7.3-eng-review for the other modes)

## Problem

The pipeline has planning agents (plan-pm, plan-em) but no code generation layer. After plan-em produces an eng plan with a populated execution table, there is no agent to write the actual code. Build mode fills this gap: a strictly scoped, platform-agnostic code writer that reads exec-table rows, scans the codebase, and writes code that passes pre-written tests — with no hardcoded platform or domain knowledge.

**Artefact to create:** `.claude/skills/eng/SKILL.md` (build mode section)

## Proposed changes

| # | What | How | Why necessary | Why ignorable | Rank |
|---|------|-----|---------------|---------------|------|
| 1 | Input contract for build mode | Input fields: `--build`, `prd-path` (path to the PRD .md containing the execution table), `rows` (list of Feature+Concern identifiers assigned to this invocation). Hard-refuse if any field is missing. Hard-refuse and report a hard failure if any assigned row has blank Execution steps — before reading any other file. Eng derives file paths; it does not accept them as input. | The execution table is the authoritative spec; blank steps mean there is no spec to write code against | Not ignorable — wrong or incomplete inputs produce ungrounded code | P1 |
| 2 | PRD summary and user approval gate | After input validation, eng reads the assigned PRD section and exec-table rows, then emits a 3–4 line summary covering: (1) what the feature is — one sentence naming the feature, its user-facing purpose, and the scope of the assigned rows; (2) how to achieve this with code — 2–3 sentences on the layers to be touched, the main structural change, and key dependencies. Presents via `AskUserQuestion`: Approve and proceed / Needs correction / I have questions. If "Needs correction" — rewrites once and re-asks. If "I have questions" — enters user interview (item 12) then re-presents. Never proceeds without explicit approval. | A misread of the PRD caught here costs one round-trip; caught after code is written costs a revert | Not ignorable — silent misreads produce confidently wrong code | P1 |
| 3 | Codebase discovery | After the approval gate, before writing any code, eng reads files most relevant to the assigned rows: existing models, API handlers, UI components, migrations. Relevance matched by concern type (e.g. "Schema migration" → scan existing migration files; "Client implementation" → scan existing UI components). Reads to understand existing patterns, naming conventions, and what must not be broken. | Without this eng writes into a vacuum and breaks existing conventions | Skippable only if the PRD explicitly marks the row as greenfield with no existing counterparts | P1 |
| 4 | Derive platform from execution concern | Maps the concern type in the exec-table row to a platform identifier: "iOS UI" / "Flutter" / "Client implementation (mobile)" → flutter; "API contract" / "Schema migration" / "Webhook" → backend; "Client implementation (web)" → web. This mapping drives which coding standards are pulled. Eng never accepts platform as an input field. | Passing platform as input breaks agnosticism when concern and platform diverge; the concern already encodes it | Not ignorable — hardcoded platform breaks platform-agnosticism | P2 |
| 5 | Coding standards pull | After deriving the platform, eng invokes the coding standards skill with the derived platform identifier and reads the result into context before writing a single line of code. Eng has no hardcoded standards of its own. | Without this eng generates code inconsistent with the project's style and conventions | Only if coding standards skill returns empty for this platform | P1 |
| 6 | Build output contract | Eng writes code to paths derived from codebase scan + exec-table. After writing, emits a structured summary: files created, files modified, platform, concern, execution table row ID for each item. The summary is the handoff artifact for review mode or a human reviewer. | Without a defined output contract the pipeline has no signal that build completed and what it produced | Not ignorable — downstream review mode and plan-em synthesis depend on this | P1 |
| 7 | Strict scope enforcement | Eng implements exactly what the assigned exec-table rows specify. It must not add features, refactor code outside the assigned row's scope, or fill gaps by assumption. Any row that is ambiguous or conflicts with the codebase scan → block, surface the specific ambiguity, wait. Does not guess. | Unscoped generation breaks plan↔code traceability and corrupts review mode's audit baseline | Not ignorable — any out-of-scope change invalidates the review | P1 |
| 8 | plan-em invocation contract | plan-em's build-mode Step 4 activates eng via the `Agent` tool. The prompt must include: `prd-path`, assigned exec-table row identifiers (Feature + Concern pairs), working branch name, and `--build`. Eng reads the full execution table from the PRD and filters to its assigned rows. The Execution steps column for each row IS the code spec — eng does not interpret beyond the steps as written. | Without a defined contract plan-em cannot reliably hand off; blank steps produce ungrounded generation | Not ignorable — the pipeline breaks at the plan→build seam | P1 |
| 9 | TDD protocol — tests before implementation | Within each feature, eng processes "Tests" concern rows BEFORE all implementation rows (API contract, schema migration, client implementation, etc.). For each "Tests" row: (1) create the test file at the derived path; (2) write syntactically valid test cases from the execution step assertions — e.g. "assert count and lastDate" becomes `expect(result.count).toBe(n)` — no `TODO` placeholders; (3) for AI/LLM-facing behavior, write eval assertions as structured objects (input, expected_shape, must_contain, must_not_contain) stored alongside tests. Implementation for a feature is only written after all its test files are in place. Tests must be runnable once implementation is complete. | TDD ensures implementation satisfies verifiable criteria. Writing tests first also surfaces row ambiguities before implementation work is sunk. | Skippable only if the PRD explicitly marks a row as prototype/throwaway | P1 |
| 10 | AHA.md learning log | Throughout the run, eng appends to `AHA.md` in the project root when any of the following occur: (a) codebase scan reveals a pattern not in the pulled coding standards; (b) an execution step cannot be implemented as written; (c) a cross-agent dependency is discovered mid-build not marked in the exec-table; (d) a non-obvious implementation decision is made; (e) a bug is found and fixed during debug mode. Format: `### [YYYY-MM-DD] <Feature — Concern>: <Summary>` with `**Issue/Learning**:` and `**Resolution**:` subsections. Append-only — never overwrites existing entries. Build summary must reference any AHA entries written during the run. | AHA.md is the project's institutional memory; without it the same errors recur and plan-em's pre-flight has no signal | Not ignorable — learnings must persist across runs | P1 |
| 11 | Debug mode — structured bug investigation loop | Activates when: (a) tests fail after implementation; (b) a written step produces a compile or runtime error; (c) a cross-agent dependency is missing or blocked. Debug cycle: `(1) Identify` — record exact failing assertion or error; `(2) Isolate` — read only the failing test file and its implementation counterpart; `(3) Hypothesize` — one specific root-cause sentence; `(4) Fix` — one targeted change within the failing step's scope only; `(5) Verify` — re-run the test or build step; `(6) Log` — write an AHA entry regardless of outcome. Max 3 cycles per issue. After 3 failed cycles: stop and escalate with a structured report (failing assertion, 3 hypotheses tried, 3 fixes applied, what information is needed). Never refactors outside failing step scope. Never applies more than one change per cycle. | The 3-cycle cap prevents runaway token burn and forces a clean escalation signal; no debug mode means broken code ships silently | Not ignorable — build mode has no integrity guarantee without it | P1 |
| 12 | User interview for unclear requirements | Eng may use `AskUserQuestion` proactively at two moments: (a) when the user selects "I have questions" at the approval gate — asks up to 3 focused questions, one at a time; (b) mid-protocol when a requirement is genuinely ambiguous and cannot be resolved from the PRD, exec-table, or codebase scan. Every question names the specific row or PRD section it concerns and offers 3–4 concrete options — no open-ended questions. Max 3 questions per invocation; if more than 3 ambiguities exist, surfaces them as a list and asks which to prioritise. After the interview, resumes protocol from where it paused. | Ambiguity resolved before code is written is cheap; ambiguity discovered after is expensive | Ignorable only if PRD and exec-table are fully unambiguous | P1 |
