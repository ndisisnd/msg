# Improvement Plan — 11-functional-mode-rigor

**Skill:** review (Functional mode)
**Change type:** Capability + rigor upgrade

## Problem

`refs/modes/functional.md` reasons about whether changed code satisfies PRD assertions by reading the diff. It never executes anything, treats every assertion the same regardless of testability, leaves the pass/warn/block rubric undefined, makes evidence optional, has no concept of "assertion N/A to this diff", silently passes assertions that were auto-generated from the same diff being reviewed (tautology), and offers no protocol for negative assertions ("X should no longer happen"). For a mode whose stated job is "verify code fits product requirements", these gaps mean a confident-sounding verdict that may have no observational basis.

## Proposed changes

| # | What | How | Why necessary | Why ignorable | Rank |
|---|------|-----|---------------|---------------|------|
| 1 | Split eval-set into `executable` vs `intent` assertions and auto-run the executable ones | Add a Step 1 classification: an assertion is `executable` if it has concrete inputs + observable outcomes (e.g. "POST /users with empty email returns 400"); otherwise `intent`. For each `executable`, generate an ephemeral script under `/tmp/review-functional-<runid>/`, run it (no network writes outside /tmp, no DB writes), parse exit code → real pass/fail. `intent` assertions follow the existing reasoning path | Without execution, Functional cannot actually test anything; it propagates confidence without evidence. This is the core gap | Skip only if the project has no runnable surface (pure type-level refactor) — Functional should then mark all executable candidates as `intent` and note the downgrade | P1 |
| 2 | Read full file context, not just diff hunks | Replace "Look for the relevant logic … in the diff" with: read each changed file in full via `Read`; widen to direct callers/callees when an assertion clearly spans a boundary | Many assertions require call-site or surrounding-logic context to verify; diff-only inspection produces false passes when the relevant guard sits 20 lines above the hunk | Trivial diffs that only touch self-contained functions don't need it — but the cost of always reading full files is small | P1 |
| 3 | Define a concrete pass/warn/block rubric with examples | Add a "Verdict rubric" subsection with explicit rules: **pass** = assertion satisfied AND evidence located; **warn** = (a) happy path implemented but error/edge path missing, (b) implementation present but intent unclear from code, (c) executable assertion failed in a way attributable to test-harness rather than code; **block** = assertion clearly violated OR required behavior absent; **n/a** = assertion does not apply to the changed surface. Include one short example per row | "Partially or ambiguously satisfied" is undefined → two runs disagree; downstream `/preflight` cannot trust the verdict | Trims if the rubric grows past a screen — keep it tight | P1 |
| 4 | Make evidence mandatory for `pass` and `block`; allow `null` only for `n/a` | Update protocol: every `pass` and `block` finding must populate `file` and `line` with the location of the satisfying or violating code. Update `refs/schema.md` to document the rule. Add a self-check step: if a `pass` has `file: null`, downgrade to `warn` with reason "no evidence located" | A verdict without a code reference is unfalsifiable; reviewers cannot audit Functional's reasoning. Mandatory evidence forces the mode to actually look | Not ignorable — this is the trust mechanism for the whole mode | P1 |
| 5 | Detect tautology when eval_set was bootstrapped from the diff | Require SKILL.md Step 3 to emit `eval_set_source: "prd" \| "diff" \| "mixed"` in the top-level output JSON. In Functional, if `eval_set_source == "diff"`, prefix the mode output with `"warning": "eval_set derived from diff — verdicts are circular by construction"` and cap the mode verdict at `warn` regardless of per-assertion outcomes. Document in `refs/schema.md` | SKILL.md Step 3 currently allows generating eval_set from the diff; checking the diff against assertions derived from the diff is a guaranteed pass and gives false confidence | Once a PRD always exists for runs (project convention enforces it), this becomes dead code — but cheap to keep | P1 |
| 6 | Add explicit handling for negative assertions | In Step 1 classification, mark assertions as `negative` when they assert removal or absence ("X should no longer happen", "Y is no longer reachable"). Verification flips: search the diff for *additions* matching the prohibited pattern → `block`; search the post-change file for *any remaining instance* of the prohibited pattern → `warn` if found, `pass` if absent. Document the inverted evidence rule (evidence = absence) | Verifying a negative from a diff is materially different from verifying a positive; without explicit handling the mode either misses violations or produces "I can't find it, so pass" non-answers | Skip only if PRDs never contain negative assertions in this project — unlikely | P2 |
| 7 | Add N/A accounting to mode output | Extend Functional's output JSON with `evaluated: N`, `n_a: M`, and a per-assertion `applicable: bool`. An assertion is `n/a` when it concerns a surface untouched by the diff (e.g. asserts billing behavior but the diff only touches auth). Update `refs/schema.md` to document the new fields. Unrelated assertions stop being silent passes; denominator is preserved | Currently, an eval_set of 10 where only 3 apply produces "10/10 pass" — meaningless. Humans and `/preflight` need to know the real denominator | Skip if every PRD assertion always applies to every change — unrealistic | P2 |
