# Improvement Plan — 7.3-eng-review

**Skill:** eng — review mode (new)
**Change type:** New capability
**Part of:** `.claude/skills/eng/SKILL.md` (three-mode skill; see 7.1-eng-build and 7.2-eng-plan for the other modes)

## Problem

Build mode has TDD and a debug loop, but no post-build quality gate. Without a structured review pass, subtle assertion gaps, untested edge cases, and code quality issues ship silently. Review mode fills this gap: an adversarial code auditor (`--review` flag) that runs the test suite, checks every exec-table assertion has a real test, proactively hunts uncovered cases, and flags code quality issues — then emits a machine-readable JSON verdict that can gate a merge.

**Artefacts to create:**
- `.claude/skills/eng/SKILL.md` (review mode section)
- `.claude/skills/eng/refs/review/template-review-output.json` — JSON output template

## Proposed changes

| # | What | How | Why necessary | Why ignorable | Rank |
|---|------|-----|---------------|---------------|------|
| 1 | Input contract for review mode | Input fields: `mode=review` (or `--review` flag), `prd-path`, `rows`. Hard-refuse if any field is missing. Eng derives file paths from the same codebase scan logic as build mode — does not accept them as input. Blank Execution steps for an assigned row are surfaced as a finding (assertion gap), not a hard failure that blocks the run. | Review mode needs to know which rows were built to know what to audit | Not ignorable — without scope the review has no baseline | P1 |
| 2 | PRD summary and user approval gate | After input validation, eng reads the assigned PRD section and exec-table rows, then emits a 3–4 line summary covering: (1) what code is being reviewed — feature name, assigned rows, file count from the codebase scan; (2) how the review will proceed — naming all four checks that will run (test execution, assertion audit, gap scan, quality scan). Presents via `AskUserQuestion`: Approve and proceed / Needs correction / I have questions. If "Needs correction" — rewrites once and re-asks. If "I have questions" — enters user interview (item 9) then re-presents. Never proceeds without explicit approval. | Confirms eng is reviewing the right scope before running any checks; a scope misread corrected here avoids a full rerun | Not ignorable — reviewing the wrong scope wastes the run and produces misleading findings | P1 |
| 3 | Platform derivation and coding standards pull | Derives platform from exec-table concern type (same mapping as build/plan modes). Invokes the coding standards skill and reads the result into context before the quality scan. Eng has no hardcoded standards. | The code quality scan must compare against the project's actual standards, not invented ones | Only if coding standards skill returns empty for this platform | P1 |
| 4 | Check 1 — Test execution | Runs the test suite scoped to files derived from the assigned rows. For each failing test, records: `test_file`, `test_name`, exact `error` message, `exec_table_row` it belongs to, and the `exec_step` the test traces to. Passing tests are recorded as counts only. | Without running tests, review mode cannot distinguish "tests were written" from "tests pass" | Not ignorable — a review that doesn't run tests is not a review | P1 |
| 5 | Check 2 — Assertion audit | Re-reads the Execution steps column for every assigned row. For each step that contains a verifiable assertion (e.g. "assert count and lastDate", "assert 401 for missing token"), checks that a real, passing test covers it. Gaps are classified: `critical` — exec-table step has no test at all; `major` — partial coverage (some assertions tested, others not); `minor` — no edge case coverage for the assertion. Each gap records: `exec_table_row`, `exec_step`, `gap` description, `severity`. | The exec-table is the contract; untested steps mean the contract is unverified | Not ignorable — assertion gaps are the primary signal that build mode is incomplete | P1 |
| 6 | Check 3 — Adversarial gap scan | Proactively identifies cases not covered by any test, even if no exec-table step explicitly requires them. Hunts: null and empty inputs, boundary values (off-by-one, max/min), concurrent access, error propagation paths, missing auth guards, unvalidated fields, missing rollback paths. Each identified gap records: `gap` description, `severity` (critical / major / minor), and the relevant file and concern. Adversarial posture: eng assumes the code has bugs and works to prove it wrong — it does not stop at the first gap found. | Tests written during TDD cover what was planned; this scan covers what was not planned but should have been | Not ignorable — this is the primary differentiator between a rubber-stamp and a real audit | P1 |
| 7 | Check 4 — Code quality scan | Reads each implementation file for the assigned rows and flags: (a) naming inconsistencies vs pulled coding standards; (b) structural issues — duplication, excessive cyclomatic complexity, unclear intent; (c) concrete refactor suggestions. Each suggestion records: `file`, `line_range`, `type` (refactor \| naming \| duplication \| complexity \| style), `severity` (critical / major / minor), `description`, and `suggested_change`. Suggestions are advisory — they do not affect the verdict. | Code quality issues not caught here compound across future builds; the quality scan makes them explicit and reviewable | Severity minor suggestions can be ignored in time-sensitive releases; the scan itself is not ignorable | P1 |
| 8 | JSON review output and verdict logic | After all four checks, eng writes the review output to `features/prd-[n]/reviews/review-<YYYYMMDD-HHmmss>.json` following `refs/review/template-review-output.json`. JSON structure: `{ run_id, prd_path, rows[], timestamp, verdict, blocker, test_results: { total, passed, failed, failures[{ test_file, test_name, error, exec_table_row, exec_step }] }, assertion_gaps[{ exec_table_row, exec_step, gap, severity }], suggestions[{ file, line_range, type, severity, description, suggested_change }], summary: { verdict, test_failures, assertion_gaps, suggestions, critical_count, major_count, minor_count, blocker } }`. Verdict rules: `fail` if any test fails or any assertion gap is `critical`; `pass` otherwise — suggestions are non-blocking. After writing the file, eng emits a short inline summary: verdict, test failure count, assertion gap count, suggestion count, and the JSON path. | The JSON is the machine-readable go/no-go signal for merge; without it the review has no durable artifact | Not ignorable — a review with no persisted output leaves no audit trail | P1 |
| 9 | User interview for unclear scope | Eng may use `AskUserQuestion` proactively at two moments: (a) when the user selects "I have questions" at the approval gate — asks up to 3 focused questions, one at a time; (b) mid-review when the scope of what was built is genuinely unclear and cannot be resolved from the PRD, exec-table, or codebase scan. Every question names the specific row or PRD section and offers 3–4 concrete options — no open-ended questions. Max 3 questions per invocation; if more than 3 ambiguities exist, surfaces them as a list and asks which to prioritise. After the interview, resumes from where it paused. | Scope ambiguity in a review produces findings against the wrong baseline — resolving it first makes the findings accurate | Ignorable only if scope is unambiguous | P1 |
