# Improvement Plan — 7.3-eng-review

**Skill:** eng — review mode (new)
**Change type:** New capability
**Part of:** `.claude/skills/eng/SKILL.md` (three-mode skill; see 7.1-eng-build and 7.2-eng-plan for the other modes)

## Problem

Build mode has TDD and a debug loop, but no post-build quality gate. Without a structured review pass, subtle assertion gaps, untested edge cases, and code quality issues ship silently. Review mode fills this gap: an adversarial code auditor (`--review` flag) that runs the test suite, checks every exec-table assertion has a real test, proactively hunts uncovered cases, and flags code quality issues — then emits a machine-readable JSON verdict that can gate a merge.

**Artefacts to create:**
- `.claude/skills/eng/SKILL.md` (review mode section)
- `.claude/skills/eng/refs/review/template-review-output.json` — JSON output template
- `.claude/skills/eng/refs/review/template-review-summary.md` — human-readable, severity-ranked findings list (each line: severity, location, issue, why to fix, gain)

## Proposed changes

| # | What | How | Why necessary | Why ignorable | Rank |
|---|------|-----|---------------|---------------|------|
| 1 | Input contract for review mode | Input fields: `--review`, `prd-path`, `rows`. Hard-refuse if any field is missing. Eng derives file paths from the same codebase scan logic as build mode — does not accept them as input. Blank Execution steps for an assigned row are surfaced as a finding (assertion gap), not a hard failure that blocks the run. | Review mode needs to know which rows were built to know what to audit | Not ignorable — without scope the review has no baseline | P1 |
| 2 | PRD summary and user approval gate | After input validation, eng reads the assigned PRD section and exec-table rows, then emits a 3–4 line summary covering: (1) what code is being reviewed — feature name, assigned rows, file count from the codebase scan; (2) how the review will proceed — naming all four checks that will run (test execution, assertion audit, gap scan, quality scan). Presents via `AskUserQuestion`: Approve and proceed / Needs correction / I have questions. If "Needs correction" — rewrites once and re-asks. If "I have questions" — enters user interview (item 9) then re-presents. Never proceeds without explicit approval. | Confirms eng is reviewing the right scope before running any checks; a scope misread corrected here avoids a full rerun | Not ignorable — reviewing the wrong scope wastes the run and produces misleading findings | P1 |
| 3 | Platform derivation and coding standards pull | Derives platform from exec-table concern type (same mapping as build/plan modes). Invokes the coding standards skill and reads the result into context before the quality scan. Eng has no hardcoded standards. | The code quality scan must compare against the project's actual standards, not invented ones | Only if coding standards skill returns empty for this platform | P1 |
| 4 | Check 1 — Test execution | Runs the test suite scoped to files derived from the assigned rows. For each failing test, records: `test_file`, `test_name`, exact `error` message, `exec_table_row` it belongs to, and the `exec_step` the test traces to. Passing tests are recorded as counts only. | Without running tests, review mode cannot distinguish "tests were written" from "tests pass" | Not ignorable — a review that doesn't run tests is not a review | P1 |
| 5 | Check 2 — Assertion audit | Re-reads the Execution steps column for every assigned row. For each step that contains a verifiable assertion (e.g. "assert count and lastDate", "assert 401 for missing token"), checks that a real, passing test covers it. Gaps are classified: `critical` — exec-table step has no test at all; `major` — partial coverage (some assertions tested, others not); `minor` — no edge case coverage for the assertion. Each gap records: `exec_table_row`, `exec_step`, `gap` description, `severity` (per the central scale), `why_fix`, and `gain`. | The exec-table is the contract; untested steps mean the contract is unverified | Not ignorable — assertion gaps are the primary signal that build mode is incomplete | P1 |
| 6 | Check 3 — Adversarial gap scan | Proactively identifies cases not covered by any test, even if no exec-table step explicitly requires them. Hunts: null and empty inputs, boundary values (off-by-one, max/min), concurrent access, error propagation paths, missing auth guards, unvalidated fields, missing rollback paths. Each identified gap is recorded in `coverage_gaps` with: `file`, `concern`, `gap` description, `severity` (per the central scale), `why_fix`, and `gain`. A `critical` coverage gap (e.g. missing auth guard, missing rollback path) **fails the verdict** — these are weighted like assertion-gap criticals, not like advisory suggestions. Adversarial posture: eng assumes the code has bugs and works to prove it wrong — it does not stop at the first gap found. | Tests written during TDD cover what was planned; this scan covers what was not planned but should have been | Not ignorable — this is the primary differentiator between a rubber-stamp and a real audit | P1 |
| 7 | Check 4 — Code quality scan | Reads each implementation file for the assigned rows and flags: (a) naming inconsistencies vs pulled coding standards; (b) structural issues — duplication, excessive cyclomatic complexity, unclear intent; (c) concrete refactor suggestions. Each suggestion records: `file`, `line_range`, `type` (refactor \| naming \| duplication \| complexity \| style), `severity` (per the central scale), `description`, `suggested_change`, `why_fix`, and `gain`. Suggestions are advisory — they do not affect the verdict. | Code quality issues not caught here compound across future builds; the quality scan makes them explicit and reviewable | Severity minor suggestions can be ignored in time-sensitive releases; the scan itself is not ignorable | P1 |
| 8 | JSON review output, ranked findings list, and verdict logic | After all four checks, eng writes the review output to `features/prd-[n]/reviews/review-<YYYYMMDD-HHmmss>.json` following `refs/review/template-review-output.json`. JSON structure: `{ run_id, prd_path, rows[], timestamp, verdict, blocker, test_results: { total, passed, failed, failures[{ test_file, test_name, error, exec_table_row, exec_step }] }, assertion_gaps[{ exec_table_row, exec_step, gap, severity, why_fix, gain }], coverage_gaps[{ file, concern, gap, severity, why_fix, gain }], suggestions[{ file, line_range, type, severity, description, suggested_change, why_fix, gain }], summary: { verdict, test_failures, assertion_gaps, coverage_gaps, suggestions, critical_count, major_count, minor_count, blocker } }`. `coverage_gaps` holds Check 3 (adversarial scan) findings. Every finding record carries `why_fix` (the reason this matters) and `gain` (what fixing it buys). `verdict` and `blocker` appear both at top level and inside `summary` **by design** — top level for a fast gate read, `summary` for a self-contained digest. Failure-output cap: record the first 10 failures with verbatim `error` text; beyond that, record a count only (mirrors "passing tests as counts only"). Verdict rules: `fail` if any test fails, **or** any `assertion_gap` is `critical`, **or** any `coverage_gap` is `critical`; `pass` otherwise — `suggestions` (Check 4) are always non-blocking. After writing the file, eng emits a **single human-readable findings list ranked worst-first** (test failures, then assertion gaps, then coverage gaps), each line showing severity, location, the issue, why to fix it, and the gain — following `refs/review/template-review-summary.md` — then the verdict and the JSON path. | The JSON is the machine-readable go/no-go signal for merge; the ranked list is the human deliverable — without them the review has no durable artifact and no actionable output | Not ignorable — a review with no persisted output leaves no audit trail | P1 |
| 9 | User interview for unclear scope | Eng may use `AskUserQuestion` proactively at two moments: (a) when the user selects "I have questions" at the approval gate — asks up to 3 focused questions, one at a time; (b) mid-review when the scope of what was built is genuinely unclear and cannot be resolved from the PRD, exec-table, or codebase scan. Every question names the specific row or PRD section and offers 3–4 concrete options — no open-ended questions. Max 3 questions per invocation; if more than 3 ambiguities exist, surfaces them as a list and asks which to prioritise. After the interview, resumes from where it paused. | Scope ambiguity in a review produces findings against the wrong baseline — resolving it first makes the findings accurate | Ignorable only if scope is unambiguous | P1 |
| 10 | Review-specific scan target (Step 2 override) | The shared Step 2 scan classifies files as Modify/Create and only reads test files when the concern type is "Tests" — neither fits review. Override it in `protocol.md`: for **every** assigned row, read both the implementation file(s) under audit **and** their corresponding test file(s), regardless of concern type. This is the single read all four checks operate on. | The assertion audit and quality scan run on whatever Step 2 loaded; if the test files for non-"Tests" rows are never read, those checks judge incomplete context | Not ignorable — wrong files in context produce wrong findings | P1 |

## Severity scale (defined once, used everywhere)

All three judgment-based checks (assertion audit, coverage gap scan, quality scan) use one scale so the ranked findings list is coherent. `protocol.md` defines it once; each check references it rather than redefining the words:

| Severity | Meaning | Verdict effect |
|----------|---------|----------------|
| `critical` | Breaks the contract or a safety guarantee: an exec-table step with no test at all, a missing auth guard, a missing rollback path | **Fails** the verdict (assertion gaps and coverage gaps only; never for advisory suggestions) |
| `major` | Partial coverage, or a real defect that is not safety-critical | Reported, does not fail the verdict |
| `minor` | Missing edge-case coverage, or a style/quality nit | Reported, does not fail the verdict |

Test failures are always treated as top-rank in the findings list and always fail the verdict.
