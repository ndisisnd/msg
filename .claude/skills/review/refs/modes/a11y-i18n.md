# review — A11y/i18n mode

**When it runs:** seventh and final in pipeline order — after Migration. **Conditional:** only runs when `active_domains[]` includes a UI-capable domain (React, Next.js, or Flutter) AND the diff touches at least one UI file in that domain (`.tsx`/`.jsx` for React/Next.js; `.dart` files containing a `Widget` build method for Flutter).

**What it checks:** missing accessibility attributes and hardcoded user-facing strings that bypass an existing i18n setup, in diff-added UI code. Static pattern checks only — **no `/cook` sub-agents**, because no accessibility or i18n concern shelf currently exists in `/cook`'s standards (see `refs/FLAG-LIST.md#extensions-with-no-domain-specific-coverage` for the equivalent gap at the language level; this is the same gap at the concern level). This mode exists so the check happens somewhere rather than nowhere, following Coverage mode's precedent of static-only analysis with no semantic fan-out.

## Trigger condition

Skip entirely (omit `a11y_i18n` from output) unless:
1. `active_domains[]` includes React, Next.js, or Flutter, **and**
2. The diff adds or modifies at least one file matching that domain's UI file pattern.

## Execution

No `/cook` sub-agents. Scan only the **added** lines of matching files (never flag pre-existing code the diff didn't touch).

### Check 1 — Missing accessibility attributes

| Domain | Pattern | Finding |
|---|---|---|
| React / Next.js | `<img` added without an `alt=` attribute on the same tag | Content image with no alt text — screen readers announce the filename or nothing. |
| React / Next.js | An added interactive element (`<button>`, `<input>`, icon-only clickable `<div onClick>`) whose only child is an icon component or SVG, with no `aria-label`, `aria-labelledby`, or visible text content | Icon-only control has no accessible name. |
| React / Next.js | An added `<input>`/`<select>`/`<textarea>` with no associated `<label>` (by `htmlFor`/`id` pairing) and no `aria-label` | Form field has no accessible name. |
| Flutter | An added custom interactive widget (wraps `GestureDetector`/`InkWell`) with no `Semantics` ancestor and no `tooltip`/`semanticLabel` param set | Custom tappable widget has no accessible label. |

### Check 2 — Hardcoded strings bypassing i18n

Only runs if the file (or the project, via a quick check for an i18n import at the top of touched files) already uses a translation mechanism — e.g. `useTranslation`/`t(...)`/`<FormattedMessage>`/`next-intl` for React/Next.js, or `easy_localization`/`intl`/`AppLocalizations` for Flutter. If the project has no i18n infra at all, skip Check 2 entirely (nothing to bypass — this is not "add i18n," it's "don't bypass the i18n you already have").

Flag an added string literal ≥ 3 words long inside JSX text content or a Flutter `Text(...)` widget that is **not** wrapped by the project's translation call, when the same file has at least one other string that **is** wrapped. (The same-file comparison is the signal that this file is expected to be translated — it avoids flagging genuinely-static strings like `console.log` debug text or internal-only admin tools with no i18n convention.)

## Output

```json
{
  "verdict": "pass" | "warn" | "block",
  "findings": [
    {
      "id": "a11y_i18n-<n>",
      "source": "a11y-i18n:static",
      "severity": "medium" | "low",
      "category": "a11y",
      "rule": "missing-alt" | "missing-aria-label" | "missing-form-label" | "missing-semantics" | "hardcoded-string",
      "message": "<description>",
      "file": "<path>",
      "line": <number>,
      "evidence": { "tool": "a11y-i18n:static", "file": "<path>", "line": <number>, "snippet": "<offending line>" },
      "suggestion": "<actionable fix>",
      "regression_of": null
    }
  ]
}
```

Findings conform to the canonical finding object (`../../shared/refs/finding-schema.md`). Severity: `medium` for missing accessibility attributes on interactive/form elements (a real usability barrier), `low` for missing alt text on decorative-adjacent content images and for hardcoded strings (a consistency nit, not a functional break) — this mode never emits `blocker`/`high`, since every check here is a heuristic pattern match, not a confirmed violation. Mode verdict: `block` never applies; `warn` if any finding exists, `pass` otherwise. Omitted from the top-level `modes` object entirely when the trigger condition doesn't match.
