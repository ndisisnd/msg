# test ‚Äî Accessibility (a11y) bucket

**When it runs:** sixth bucket in `--sequential` order ‚Äî after Load; by default runs concurrently as its own subagent.

**What it checks:** WCAG 2.1/2.2 violations across pages or components using the detected accessibility audit runner.

## Execution

Reads `a11y_runner` from the Step 1 fingerprint ‚Äî does not re-detect.

### Step 1 ‚Äî Guard

If `a11y_runner` is `null`: emit `pass_with_warnings` with note `"No accessibility runner detected ‚Äî a11y bucket skipped."` and return immediately.

Recognised runners (detection order):

| Runner | Detection signal | Default command |
|--------|-----------------|-----------------|
| axe-core CLI | `@axe-core/cli` in devDeps or `axe` in scripts | `npx axe <urls> --reporter json` |
| axe-playwright | `axe-playwright` in devDeps | run via existing Playwright config with axe integration |
| jest-axe | `jest-axe` in devDeps | `npx jest --testPathPattern=a11y` (or closest match) |
| pa11y | `pa11y` or `pa11y-ci` in devDeps or scripts | `npx pa11y-ci` (uses `.pa11yci` config if present) |
| Lighthouse (a11y) | `lighthouse` or `lhci` in devDeps / scripts | `npx lhci autorun --collect.settings.onlyCategories=accessibility` |

Use the first match found.

### Step 2 ‚Äî Resolve target URLs / components

Determine what to audit:

1. **Config file** ‚Äî if the runner has a config (`.pa11yci`, `lighthouserc.*`, axe config), read the URLs/paths from it.
2. **Sitemap / routes** ‚Äî if no config, look for `sitemap.xml`, a routes file, or a `baseURL` in `playwright.config.*`. Extract up to 20 pages, starting with `/` (root).
3. **Storybook** ‚Äî if `storybook` is in scripts and the runner is jest-axe or axe-playwright, run against the Storybook build instead.
4. **Fallback** ‚Äî if no URLs can be resolved, emit `pass_with_warnings` with note `"No audit targets found ‚Äî provide URLs via runner config."` and return.

Emit: `A11y targets: <N> pages/components.`

### Step 3 ‚Äî Run

Execute `a11y_runner.command` against the resolved targets. Capture stdout, stderr, exit code, and any JSON report output.

- **Exit 0, zero violations** ‚Üí verdict `pass`.
- **Exit 0 / non-zero, violations present** ‚Üí verdict `fail` for `critical` or `serious` violations; `pass_with_warnings` for `moderate` or `minor` only.
- **Non-zero exit, runner crash** ‚Üí verdict `pass_with_warnings` with note `"A11y runner failed to start ‚Äî results unreliable."`.

### Step 4 ‚Äî Parse violations

Map runner-native severity to the canonical finding severity (`../../../shared/refs/finding-schema.md`):

| Runner severity | Finding severity |
|----------------|-----------------|
| critical / serious | `high` |
| moderate | `medium` |
| minor / best-practice | `medium` |

For each violation, extract:

- `file` ‚Äî page URL or component file path
- `line` ‚Äî DOM node selector or `null` (line numbers not applicable for rendered violations)
- `rule` ‚Äî WCAG criterion + rule ID (e.g. `"WCAG 1.1.1 ‚Äî image-alt"`)
- `message` ‚Äî human-readable description of the violation
- `repro` ‚Äî command or URL to reproduce the audit on just this page
- `suggestion` ‚Äî fix description from the runner (axe and Lighthouse both provide these)
- artifact ‚Äî path to a screenshot of the audited page at the time of the violation, if the runner produced one (goes in `evidence.file`)

### Step 5 ‚Äî Deduplicate

If the same rule fires on multiple pages, collapse into a single finding with `message` updated to `"<rule> ‚Äî N pages affected (e.g. /, /about, /contact)"`. Keep individual findings for distinct selectors on the same page.

## Error handling

A bucket-level error never stops other buckets. All errors produce a `pass_with_warnings` verdict (not `fail`) so a broken audit environment does not falsely block a merge.

| Error condition | Verdict | Note in output |
|----------------|---------|----------------|
| Runner binary not found / not installed | `pass_with_warnings` | `"<runner> not found ‚Äî install it or add it to devDependencies."` |
| Runner crashes on startup (non-zero, no report) | `pass_with_warnings` | `"A11y runner failed to start ‚Äî results unreliable."` Include stderr excerpt (max 5 lines). |
| Runner crashes mid-run (partial report exists) | Emit partial findings; overall `pass_with_warnings` | `"A11y runner crashed after <N> of <M> targets ‚Äî partial results only."` |
| Target URL unreachable (connection refused / 404) | Skip that URL; continue; `pass_with_warnings` if ALL targets fail | `"Could not reach <url> ‚Äî skipped."` |
| Network timeout (runner hangs > 120 s per target) | Kill the runner for that target; continue | `"Timed out auditing <url> after 120 s ‚Äî skipped."` |
| Config parse error | `pass_with_warnings` | `"Could not parse runner config at <path>: <error>."` Fall back to CLI defaults. |
| No audit targets resolved (Step 2 fallback) | `pass_with_warnings` | `"No audit targets found ‚Äî provide URLs via runner config."` |
| Zero pages audited successfully | `pass_with_warnings` | `"All audit targets failed or timed out ‚Äî no findings."` |

**Partial results rule:** if at least one target was audited successfully, emit whatever findings exist and set the verdict based on those findings (not `pass_with_warnings` due to errors alone). Attach a top-level `"errors"` array to the bucket output listing each skipped/failed target and its reason.

## Output

```json
{
  "verdict": "pass" | "pass_with_warnings" | "fail",
  "bucket": "a11y",
  "errors": [
    { "target": "<url or component>", "reason": "<error description>" }
  ],
  "runner": "<a11y_runner.name>",
  "command": "<command executed>",
  "targets_audited": <number of pages/components>,
  "totals": { "critical": 0, "serious": 0, "moderate": 0, "minor": 0 },
  "findings": [
    {
      "id": "a11y-<n>",
      "source": "a11y",
      "severity": "high" | "medium",
      "category": "a11y",
      "file": "<page URL or component path>",
      "line": null,
      "rule": "<WCAG criterion ‚Äî rule id>",
      "message": "<violation description>",
      "evidence": {
        "tool": "<a11y_runner.name>",
        "file": "<screenshot path or null>",
        "line": null,
        "snippet": "<violation description>"
      },
      "suggestion": "<fix description or null>",
      "repro": "<audit command or URL>",
      "regression_of": null
    }
  ]
}
```

`fail` if any `critical` or `serious` WCAG violation is found. `pass_with_warnings` if only `moderate`/`minor` violations, no targets found, or runner crashed. `pass` if zero violations.
