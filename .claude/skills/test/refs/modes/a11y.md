# test — Accessibility (a11y) bucket

**When it runs:** sixth bucket in `--sequential` order — after Load; by default runs concurrently as its own subagent.

**What it checks:** WCAG 2.1/2.2 violations across pages or components using the detected accessibility audit runner.

## Execution

Guard, bucket-error rule, and output envelope: see `_common.md`. `a11y_runner` (name, command, e.g. axe-core CLI / axe-playwright / jest-axe / pa11y / Lighthouse) comes from the Step 1 fingerprint — this bucket does not re-detect.

### Step 1 — Guard

Per `_common.md`: if `a11y_runner` is `null`, emit `pass_with_warnings` with note `"No accessibility runner detected — a11y bucket skipped."` and return immediately.

### Step 2 — Resolve target URLs / components

Determine what to audit:

1. **Config file** — if the runner has a config (`.pa11yci`, `lighthouserc.*`, axe config), read the URLs/paths from it.
2. **Sitemap / routes** — if no config, look for `sitemap.xml`, a routes file, or a `baseURL` in `playwright.config.*`. Extract up to 20 pages, starting with `/` (root).
3. **Storybook** — if `storybook` is in scripts and the runner is jest-axe or axe-playwright, run against the Storybook build instead.
4. **Fallback** — if no URLs can be resolved, emit `pass_with_warnings` with note `"No audit targets found — provide URLs via runner config."` and return.

Emit: `A11y targets: <N> pages/components.`

### Step 3 — Run

Execute `a11y_runner.command` against the resolved targets. Capture stdout, stderr, exit code, and any JSON report output.

- **Exit 0, zero violations** → verdict `pass`.
- **Exit 0 / non-zero, violations present** → verdict `fail` for `critical` or `serious` violations; `pass_with_warnings` for `moderate` or `minor` only.
- **Non-zero exit, runner crash** → verdict `pass_with_warnings` with note `"A11y runner failed to start — results unreliable."`.

### Step 4 — Parse violations

Map runner-native severity to the canonical finding severity (`../../../shared/refs/finding-schema.md`):

| Runner severity | Finding severity |
|----------------|-----------------|
| critical / serious | `high` |
| moderate | `medium` |
| minor / best-practice | `medium` |

For each violation, extract:

- `file` — page URL or component file path
- `line` — DOM node selector or `null` (line numbers not applicable for rendered violations)
- `rule` — WCAG criterion + rule ID (e.g. `"WCAG 1.1.1 — image-alt"`)
- `message` — human-readable description of the violation
- `repro` — command or URL to reproduce the audit on just this page
- `suggestion` — fix description from the runner (axe and Lighthouse both provide these)
- artifact — path to a screenshot of the audited page at the time of the violation, if the runner produced one (goes in `evidence.file`)

### Step 5 — Deduplicate

If the same rule fires on multiple pages, collapse into a single finding with `message` updated to `"<rule> — N pages affected (e.g. /, /about, /contact)"`. Keep individual findings for distinct selectors on the same page.

## Error handling

Applies `_common.md`'s bucket-error rule (every error → `pass_with_warnings`, never `fail`) with these a11y-specific cases:

| Error condition | Verdict | Note in output |
|----------------|---------|----------------|
| Runner binary not found / not installed | `pass_with_warnings` | `"<runner> not found — install it or add it to devDependencies."` |
| Runner crashes on startup (non-zero, no report) | `pass_with_warnings` | `"A11y runner failed to start — results unreliable."` Include stderr excerpt (max 5 lines). |
| Runner crashes mid-run (partial report exists) | Emit partial findings; overall `pass_with_warnings` | `"A11y runner crashed after <N> of <M> targets — partial results only."` |
| Target URL unreachable (connection refused / 404) | Skip that URL; continue; `pass_with_warnings` if ALL targets fail | `"Could not reach <url> — skipped."` |
| Network timeout (runner hangs > 120 s per target) | Kill the runner for that target; continue | `"Timed out auditing <url> after 120 s — skipped."` |
| Config parse error | `pass_with_warnings` | `"Could not parse runner config at <path>: <error>."` Fall back to CLI defaults. |
| No audit targets resolved (Step 2 fallback) | `pass_with_warnings` | `"No audit targets found — provide URLs via runner config."` |
| Zero pages audited successfully | `pass_with_warnings` | `"All audit targets failed or timed out — no findings."` |

**Partial results rule:** if at least one target was audited successfully, emit whatever findings exist and set the verdict based on those findings (not `pass_with_warnings` due to errors alone). Attach a top-level `"errors"` array to the bucket output listing each skipped/failed target and its reason.

## Output

Envelope + finding shape per `_common.md`. Bucket fields:

```json
"errors": [ { "target": "<url or component>", "reason": "<error description>" } ],
"runner": "<a11y_runner.name>", "command": "<command executed>",
"targets_audited": <number of pages/components>,
"totals": { "critical": 0, "serious": 0, "moderate": 0, "minor": 0 }
```

Findings: category/source `a11y`; `rule` = `"<WCAG criterion — rule id>"`, `evidence.file` = page screenshot or null.

`fail` if any `critical` or `serious` WCAG violation is found. `pass_with_warnings` if only `moderate`/`minor` violations, no targets found, or runner crashed. `pass` if zero violations.
